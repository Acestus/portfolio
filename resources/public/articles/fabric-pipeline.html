1. Medallion Architecture

The Bronze, Silver, and Gold layers are not buzzwords here; they divide responsibility. Bronze is append-only raw ingestion with minimal assumptions. Silver is where schemas become trustworthy: timestamps are typed, duplicate records are handled, payloads are normalized, and low-quality events are quarantined instead of silently polluting analytics. Gold is intentionally small and business-shaped, optimized for report refresh and semantic clarity rather than event fidelity.

LayerReal ResponsibilityTypical Storage Shape
BronzeLand raw transaction events exactly as producedAppend-only ingestion table with schema-on-read tolerance
SilverParse JSON, type columns, deduplicate, enforce quality checksKQL-managed analytical table with update policy or transformation function
GoldPre-aggregate metrics for BI consumptionLakehouse Delta tables consumed by semantic model

2. Event Hub Ingestion

The architectural shift is important: instead of application code writing synchronously into SQL tables, services emit events to Event Hub. That decouples transactional paths from analytics paths. Eventstream then becomes the low-friction bridge into Fabric, giving you fan-in ingestion, observability, and a clear handoff between application teams and data platform ownership.

In the SPA, the animated pipeline and activity log make that architecture visible. A simple timer advances the active stage every two seconds, while sample operations such as transaction_raw, update_policy, notebook_tfm_gld, and semantic_refresh reinforce that data movement is a chain of independent, observable steps.

3. KQL Update Policies

KQL update policies are the key to making Eventhouse useful as more than a raw landing zone. Instead of waiting for batch jobs to parse logs later, the raw table can trigger transformation into a typed analytical table as events arrive. That keeps the ingest path cheap while still giving downstream consumers clean columns.

.create-or-alter function with (folder = 'transactions') TransformTransactions() { transaction_raw | extend event_ts = todatetime(raw_timestamp) | extend tenant_id = tostring(payload.tenantId) | extend order_id = tostring(payload.orderId) | extend amount = todouble(payload.amount) | extend currency = tostring(payload.currency) | extend payment_status = tostring(payload.paymentStatus) | extend channel = tostring(payload.channel) | project event_ts, tenant_id, order_id, amount, currency, payment_status, channel, raw_payload } .alter table transaction_silver policy update @'[{ "IsEnabled": true, "Source": "transaction_raw", "Query": "TransformTransactions()", "IsTransactional": false, "PropagateIngestionProperties": true }]'

This is exactly the kind of transformation boundary that scales well in production: raw events stay available for replay and auditing, but analysts and notebooks consume a curated schema immediately.

4. Gold Layer & DAX/TMDL

The Gold layer should contain opinionated business outputs, not a second Silver layer with prettier names. In this design, notebooks aggregate hourly and daily transaction facts into two gold tables, then the semantic model exposes measures for reporting tools. TMDL matters because semantic definitions become source-controlled artifacts rather than portal-only configuration.

Revenue := SUMX ( 'gld_transactions_hourly', 'gld_transactions_hourly'[approved_amount] ) Approval Rate := DIVIDE ( [Approved Transactions], [Total Transactions] )

That separation is why the dashboard can talk about 14 DAX measures without turning the ingest tier into a BI engine. Eventhouse handles stream transformation, notebooks handle aggregation, and the semantic model handles business logic and presentation semantics.

5. Production Lessons

Three practical lessons show up in pipelines like this. First, scheduling workarounds matter: when a desired trigger is not natively available in exactly the right place, teams often bridge with pipeline orchestration, notebook scheduling, or a lightweight automation layer rather than forcing synchronous coupling back into the applications. Second, Git integration matters at the workspace level; treating notebooks, semantic definitions, and pipeline metadata as code makes promotion and rollback much safer. Third, privileged access still needs discipline. Admin actions in Fabric and Azure should be performed through PIM-backed elevation, not standing rights.

The result is a cleaner architecture than direct SQL writes ever allowed. Application services emit facts. The platform translates, shapes, and publishes them. Analytics becomes a product of the event stream, not a side effect of OLTP code.
