1. SKU Selection and Right-Sizing

Azure pricing is SKU-driven. Every resource has a tier that determines its capabilities and cost. The challenge is not finding the cheapest option — it is matching the tier to the actual workload profile. Over-provisioning wastes budget; under-provisioning causes outages and emergency scale-ups that cost more than the original tier would have.

The SPA models four service categories — Compute, Storage, Database, and Networking — each with three tiers. In production, this maps directly to decisions like choosing between an App Service B1 ($13/mo) for a low-traffic web frontend and an S1 ($73/mo) for a service that needs autoscale and deployment slots.

2. Cost Management Patterns

Cost management in Azure is not a monthly review — it is a set of automated guardrails that catch overspend before it happens. The most effective controls are tagging for chargeback, budget alerts with action groups, weekly Advisor reviews for right-sizing, and Reserved Instance commitments for stable workloads.

Quick Billing Check

az consumption usage list \ --start-date 2025-01-01 \ --end-date 2025-01-31 \ --query "[].{Resource:instanceName, Cost:pretaxCost, Currency:currency}" \ --output table

3. JML Cost Impact

Workforce lifecycle events — Joiner, Mover, Leaver — create and retire cloud costs. A new hire triggers compute provisioning, license assignment, and storage allocation. A transfer reallocates resources between cost centers. Offboarding reclaims licenses and deallocates per-user resources. The SPA models four scenarios to show how headcount changes ripple through Azure spend.

The most significant cost driver is often licensing, not infrastructure. A Microsoft 365 Copilot add-on at $30/user/month on top of an E3/E5 base license can exceed the per-user Azure infrastructure cost for many workloads. JML automation ensures licenses are assigned at onboarding and reclaimed at offboarding — the gap between those two events is where license waste accumulates.

4. Tag Enforcement with Modify Policies

Tags are enforced via Azure Policy modify-effect policies that auto-append missing required tags during resource creation. This is non-blocking governance: deployments succeed, but tags are guaranteed to be present.

How Modify Works

  1. A resource group is created without the required CostCenter tag
  2. The policy evaluates the resource at creation time
  3. The modify effect adds the tag with a default value (inherited from subscription or resource group)
  4. The resource is created with the tag already present

These are non-blocking — they add missing tags at resource creation time without denying the deployment. Existing resources missing tags can be remediated with a remediation task or the PowerShell bulk tag merge script.

Bulk Tag Remediation

# Query resource groups with missing tags, merge new tags without overwriting existing ones $rg = Get-AzResourceGroup | Where-Object { -not $_.Tags.ContainsKey('CostCenter') } foreach ($group in $rg) { $existing = $group.Tags ?? @{} $existing['CostCenter'] = 'inherited-from-subscription' Set-AzResourceGroup -Name $group.ResourceGroupName -Tag $existing }

5. CAF Naming Conventions

The Cloud Adoption Framework naming convention ensures every resource is identifiable by type, project, environment, and region at a glance. The Azure Naming Tool enforces the pattern {type}-{project}-{environment}-{region} at provisioning time. Names invented outside this convention are blocked at PR gates and cannot be promoted to staging or production.

Deny Policies — Network Security

Two critical deny policies enforce network security boundaries. They block resource creation at the ARM layer, forcing all internet-facing traffic through controlled ingress points (Application Gateway, Azure Firewall, or Load Balancer).
