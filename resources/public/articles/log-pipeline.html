Real-Time Log Pipeline — Event Hub to Eventhouse

Animated visualization of a real-time log ingestion pipeline. The SPA makes that flow visible with a live log stream and KQL panel.

1. Architecture Overview

The architecture solves a classic scaling mistake: using application databases as log sinks. In the old world, operational services often pushed diagnostic rows straight into SQL tables. That creates contention, schema coupling, and poor separation between OLTP and observability. In the event-driven version, applications emit facts, Event Hub absorbs throughput, and downstream systems parse and aggregate independently.

The SPA mirrors that design in miniature. The stage row is rendered from a data structure of pipeline nodes, and a timer prepends new sample events into the stream while trimming the list to the latest 15 lines. That means the page behaves like an active log console without requiring a backend.

2. KQL Update Policies

Real-time value comes from parsing near the ingestion point. Raw logs are still useful for replay, but analytical questions need typed columns: timestamps, app names, levels, HTTP status, and message body. KQL transformation functions are ideal here because they express parsing as readable, versionable query logic.

.create-or-alter function TransformRawLogs() { transaction_raw | extend parsed_ts = todatetime(raw_timestamp) | extend app_name = extract('app=([\\w-]+)', 1, raw_message) | extend log_level = extract('level=(\\w+)', 1, raw_message) | extend http_status = toint(extract('→ (\\d+)', 1, raw_message)) | project parsed_ts, app_name, log_level, http_status, raw_message }

A production implementation would usually pair that function with an update policy on a parsed table so the raw table remains authoritative and the parsed table stays continuously queryable. That pattern gives you both forensic retention and analytical convenience.

3. Gold Lakehouse Aggregation

Not every query belongs in Eventhouse. Once logs are parsed, a PySpark notebook can aggregate them into gold datasets such as hourly request counts, failure rates by service, or P95 latency bands. That is the right place to denormalize for BI, because the transformations are explicit and batch-oriented instead of hidden inside dashboard formulas.

df = spark.read.table('transaction_parsed') (df.groupBy('app_name', window('parsed_ts', '1 hour')) .agg(count('*').alias('events'), avg('http_status').alias('avg_status')) .write.mode('overwrite') .format('delta') .saveAsTable('gld_app_hourly'))

4. Monitoring & Observability

The page's live stream is more than decoration. It illustrates the observability feedback loop: ingest, parse, classify, and watch for anomalies. The demo color-codes INFO, WARN, and ERROR records, which is exactly the kind of lightweight signal that helps an operator decide whether a red metric is a transient spike or a systemic issue.

In a real estate, you would pair these visuals with lag monitoring, malformed message counts, Event Hub throughput metrics, and alerting on transformation failures. The architectural point remains the same: observability pipelines need observability too.

5. From SQL Inserts to Event-Driven

The most important shift is organizational as much as technical. Once logs become events instead of table inserts, applications no longer need to know how analytics stores are modeled. They only need to emit durable, well-formed facts. The platform then owns enrichment, schema evolution, retention, and downstream publication.

That is why this SPA works as a portfolio piece. It does not just show pretty boxes. It tells the story of decoupling, contract design, and operational maturity—exactly the reasons teams move from brittle insert-based telemetry to streaming pipelines.
