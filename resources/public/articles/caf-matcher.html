CAF Naming Convention

The puzzle is built around a practical naming grammar, not arbitrary strings. Resource names encode type, project, environment, region, and instance. That makes them readable to humans while still predictable enough for scripts, governance policies, and support runbooks.

The Naming Pattern

The article-worthy detail is that the game teaches structure, not memorization. Players are effectively parsing a naming DSL. Once you recognize that usw2 is West US 2 and ctl signals a control identity or control-plane role, the string stops being noise and starts becoming an inventory record.

{type}-{project}-{env}-{region}-{instance} func-project-dev-usw2-001 st-project-dev-usw2-001 umi-project-dev-usw2-ctl

Because the source data is a vector of maps, the game can separate display text from semantics. A round is generated by taking five random pairs, then independently shuffling the resource column and CAF-name column. That means every round is small enough to read yet different enough to stay interesting.

Why Naming Matters

Good naming reduces cognitive load during incidents, deployments, and audits. If a platform engineer sees kv-skpad-prd-usw2-001, they can infer service class, workload, environment, geography, and multiplicity without opening Azure Portal. That is exactly why naming standards belong in policy, IaC, dashboards, and onboarding material.

The game also quietly shows why inconsistent naming is expensive. Every mismatch forces extra lookup steps. In real systems that means harder chargeback, weaker troubleshooting, and brittle automation because regex, tag rules, and deployment scopes all become less trustworthy.

Implementation in ClojureScript

This SPA renders entirely on canvas, so it must convert browser click coordinates into game coordinates. The code reads the canvas bounding rectangle, derives X and Y scale factors, and maps pointer position back into the 480×400 game world before deciding which row was selected.

rect = canvas.getBoundingClientRect() scale-x = W / rect.width scale-y = H / rect.height mx = (clientX - rect.left) * scale-x my = (clientY - rect.top) * scale-y

That one step is the difference between a toy and a robust canvas UI. It preserves hit accuracy regardless of CSS scaling, making the puzzle usable on phones and tablets without changing the logical layout or introducing a second rendering path.
