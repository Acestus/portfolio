1. Trunk-Based Delivery

The pipeline follows a Dave Farley style of trunk-based delivery: small changes integrated continuously, short-lived branches, and deployment logic exercised every day rather than saved for a release train. That matters more for infrastructure than many teams realize. Rare deployments make IaC look deterministic on paper and unpredictable in practice.

The SPA's animated flow reflects that philosophy. A timer cycles the active stage, visually emphasizing that delivery is a repeatable pipeline, not a heroic manual ceremony performed once a month.

2. OIDC Federation

The security model is one of the strongest parts of the design. GitHub Actions authenticates to Azure through OIDC federation, which means no long-lived client secrets need to be stored in the repository or runner settings. The workflow requests a short-lived token bound to the repo, environment, and job context, and Azure validates the trust relationship before issuing access.

That reduces secret rotation overhead and shrinks the blast radius of CI compromise. More importantly, it aligns the delivery story with the platform story: if we tell application teams not to keep secrets in pipelines, our own infrastructure pipeline should model that standard.

3. Azure Verified Modules (AVM-first approach)

Choosing AVM first is an engineering strategy, not a branding preference. Verified modules encode known-good inputs, outputs, and deployment shapes for common Azure resources. That means teams spend less effort rewriting wrappers for storage accounts, identities, key vaults, or eventing resources and more effort composing higher-level platform patterns.

In practice, AVM-first usually leads to thinner custom Bicep: more composition, fewer bespoke resource definitions, and easier upgrades when Azure APIs evolve.

4. Deployment Stacks

Deployment Stacks solve a chronic IaC problem: lifecycle ambiguity. When resources are associated with a stack, deletion, drift reasoning, and scope ownership become more explicit. For platform teams managing many resource groups and environments, that is a major operational advantage over loose template deployments.

5. CAF Naming Convention

The naming scheme is explicit and automatable. It encodes resource class, project, environment, region, and instance in a compact pattern that works equally well for humans, scripts, and policy checks.

{type}-{project}-{env}-{region}-{instance} func-skpad-dev-usw2-001 st-skpad-dev-usw2-001 kv-skpad-dev-usw2-001 umi-skpad-dev-usw2-ctl

The final segment is especially useful. Numeric instances distinguish deployable units, while suffixes like ctl or dat can identify role or purpose. That small convention pays off everywhere from IAM scoping to support triage.

6. Dual-UMI Pattern

A dual-UMI pattern separates control-plane and data-plane identity concerns. One user-assigned managed identity is used for deployment and resource orchestration. A second identity is used by the workload itself for runtime access to services such as Key Vault, Storage, or Event Hub. That boundary keeps least privilege enforceable and makes auditing easier.

In other words: the identity that creates resources should not automatically be the identity that reads production data. Mature IaC pipelines encode that distinction from day one.
